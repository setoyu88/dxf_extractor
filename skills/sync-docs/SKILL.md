## ユーザー入力

```text
$ARGUMENTS
```

引数が指定されている場合は、対象ディレクトリ・ファイルの絞り込みに使用する。

---

## 目的

ソースコードに変更が加わった際、ドキュメント（README・仕様書・スキーマ定義等）が古い記述のまま放置されることを防ぐ。

**ソースコードが唯一の正**であり、ドキュメントをソースコードに合わせて更新する。  
ソースコードの動作・ロジック・インターフェースには一切変更を加えない。

---

## Step 1: プロジェクト構造の自動検出

### 1-1. ソースコードの発見

以下の優先順位でソースディレクトリを特定する。

**引数指定がある場合**: 引数で指定されたパスを最優先とする。

**引数指定がない場合**: 以下の順で自動検出する。

| 優先度 | 検索対象 | 例 |
|--------|---------|-----|
| 1 | `src/` 配下のソースファイル | `src/**/*.py`, `src/**/*.ts` |
| 2 | `lib/`, `app/`, `pkg/` 配下 | `lib/**/*.go`, `app/**/*.rb` |
| 3 | プロジェクトルート直下のソースファイル | `*.py`, `*.js` |

**言語検出**: 拡張子の出現頻度からプロジェクトの主言語を判定する。

| 言語 | 対象拡張子 | テストコマンド候補 |
|------|-----------|-----------------|
| Python | `.py` | `uv run pytest` / `python -m pytest` / `pytest` |
| TypeScript/JavaScript | `.ts`, `.js`, `.tsx`, `.jsx` | `npm test` / `yarn test` / `pnpm test` |
| Go | `.go` | `go test ./...` |
| Rust | `.rs` | `cargo test` |
| Java | `.java` | `mvn test` / `gradle test` |
| Ruby | `.rb` | `bundle exec rspec` |
| C# | `.cs` | `dotnet test` |

テストコマンドは以下の設定ファイルから優先的に決定する:  
`pyproject.toml` → `package.json` → `Makefile` → `Cargo.toml` → `go.mod`

### 1-2. ドキュメントの発見

**引数指定がある場合**: 指定ファイルのみを対象とする。

**引数指定がない場合**: 以下を対象として自動検出する。

| 優先度 | 対象パターン | 種別 |
|--------|------------|------|
| 1 | `README.md`, `README.rst` | プロジェクト概要 |
| 2 | `docs/**/*.md` | ドキュメントディレクトリ |
| 3 | `specs/**/*.md` | 仕様書ディレクトリ |
| 4 | `spec/**/*.md` | 仕様書ディレクトリ（別名） |
| 5 | `CHANGELOG.md`, `CONTRIBUTING.md` | その他ドキュメント |

codegraphが利用可能であれば `codegraph_status` で確認し、`codegraph_files` でファイル一覧を補完する。

---

## Step 2: ソースコードの現行実装を読み取る

### 2-1. codegraph を活用した実装把握

codegraphインデックスが利用可能な場合（`codegraph_status` で確認）:

1. `codegraph_explore` でメインモジュール・公開APIを把握する
2. `codegraph_context` で主要な関数・クラスの仕様を確認する
3. `codegraph_search` で特定のシンボルや機能を検索する

codegraphが利用不可の場合: Glob + Read で直接ソースファイルを読み取る。

### 2-2. 抽出すべき実装情報

以下の情報をソースコードから体系的に抽出する:

| カテゴリ | 抽出すべき情報 |
|---------|--------------|
| **公開API** | 関数名・メソッド名・シグネチャ・引数・戻り値の型 |
| **データモデル** | クラス・構造体のフィールド名・型・デフォルト値 |
| **CLIインターフェース** | コマンド名・オプション・引数・終了コード |
| **設定** | 設定項目名・型・デフォルト値・必須/任意 |
| **対応機能** | サポートする入力形式・出力形式・アルゴリズムの種別 |
| **エラー処理** | 例外クラス・エラーコード・エラーメッセージ |
| **依存関係** | 外部ライブラリ・バージョン制約 |

### 2-3. テストコードを補足証拠として参照

`tests/` または `test/` 配下のテストファイルを読み、実装の振る舞いの補足証拠とする（テストコードの変更は禁止）。

---

## Step 3: ドキュメントの現行記述を読み取る

Step 1-2 で特定したドキュメントファイルをすべて読み込む。  
ファイルが大きい場合は対象セクションに絞って読む。

各ドキュメントについて以下を記録する:
- ファイルパス
- 記載されている機能・API・型・設定の一覧
- 最終更新日時（git logで確認可能な場合）

---

## Step 4: 齟齬を特定する

Step 2 と Step 3 の内容を照合し、以下の齟齬パターンを検出する。

| パターン | 説明 | 例 |
|---------|------|---|
| **誤記** | ドキュメントが実在しないフィールド・型・オプションを記載 | 削除済みフィールドの記述が残っている |
| **欠落** | 実装が対応しているが、ドキュメントに記載がない | 新機能追加後にREADMEが未更新 |
| **実装詳細の誤り** | 算出方法・アルゴリズムの説明が実装と異なる | 「ライブラリAを使用」と書いてあるが実際は独自実装 |
| **古いパス・コマンド** | 存在しないファイルパス・変更されたコマンド名への参照 | 廃止されたCLIオプションの記述 |
| **型の不整合** | ドキュメントの型定義がモデルの実装と異なる | `int` と書いてあるが実際は `str` |
| **デフォルト値の誤り** | ドキュメントのデフォルト値が実装と異なる | `default=True` と書いてあるが実際は `False` |

各齟齬について以下を記録する:
- ファイルパスと該当箇所（行番号）
- 現在のドキュメント記述
- 正しい実装に基づく記述
- 齟齬パターンの分類

---

## Step 5: ドキュメントを更新する

### 更新ルール

1. **ソースコードを正とする**: ドキュメントの記述がソースコードと異なる場合は、ソースコードに合わせてドキュメントを修正する
2. **動作に影響する変更は禁止**: ソースコードへの変更はコメント追加のみ許容する
3. **最小変更の原則**: 齟齬のある箇所のみ修正し、無関係な文体・構成の変更は行わない
4. **Edit ツールを使用**: ファイル全体を書き直さず、差分箇所のみ `Edit` ツールで修正する
5. **言語を維持**: ドキュメントの既存言語（日本語/英語）を変更しない

### 更新禁止事項

- ソースコードのロジック・インターフェース・動作を変更してはならない
- ドキュメントの構成・見出し階層を大幅に変更してはならない（修正箇所のみ変更）
- `tasks.md` のタスク完了状態（`[x]` / `[ ]`）を変更してはならない
- `CHANGELOG.md` の既存エントリを変更してはならない（追記のみ許容）

---

## Step 6: テストを実行して動作への影響がないことを確認する

Step 1-1 で検出したテストコマンドを実行する。

```
<検出されたテストコマンド> -q  # またはサイレントオプション相当
```

**判定基準**:
- 全テストが通過 → ドキュメント更新完了
- テスト失敗 → ドキュメント更新の副作用ではなく、実装に起因する問題として報告する（ドキュメント変更はロールバックしない）

---

## 出力形式

### 差分サマリー（必須出力）

以下の形式でターミナルに出力する:

```
== ドキュメント同期サマリー ==

README.md
  - [修正] <セクション名>: <変更前の記述> → <変更後の記述>
  - [追記] <セクション名>: <追記した内容>
  - [変更なし]

docs/api.md
  - [修正] <フィールド名>の型: <変更前> → <変更後>

== 合計 ==
修正: N箇所 / 追記: N箇所 / 変更なし: Nファイル
テスト: PASS（Nテスト） / SKIP（テストなし）
```

齟齬が1件も見つからなかった場合:
```
== ドキュメント同期サマリー ==
ドキュメントとソースコードの間に齟齬は検出されませんでした。
```

---

## 動作原則

- **ソースコードは変更しない**: コメント追加は許容するが、ロジック・インターフェースに影響する変更は厳禁
- **ドキュメント構造を壊さない**: 既存の見出し・セクション構成を尊重し、差分箇所のみ修正
- **推測で書かない**: ソースコードから確認できない情報をドキュメントに追記しない
- **テスト失敗は報告のみ**: テスト失敗があってもドキュメント変更はロールバックしない（失敗は実装の問題として報告）
